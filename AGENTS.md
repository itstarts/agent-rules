# AGENTS.md — 全局通用工程纪律

> 本文件是跨项目通用的工作约定（global baseline），只包含与技术栈无关的工程纪律与协作规则。
> 栈相关、业务相关强约束的完整清单与具体规则（DB/SQL、MQ、资金、权限、脱敏等）见项目级 `AGENTS.md` / `CLAUDE.md`；本文件仅在任务分级等处引用其中少量类别作为通用判断锚点。
> 文中“Agent”指执行任务的助手，不绑定具体运行时。

## 适用范围与优先级

- 本文件适用于当前工作区及其所有子项目；子目录存在更具体的 `AGENTS.md` 时，优先遵循子目录规则。
- 指令优先级：当前用户明确指令 > 项目级规则 > 本全局规则 > 默认系统行为。
- 项目级规则可细化流程选择，但不得放宽以下硬约束：无验证证据时不得声称“已完成”“已修复”或“测试通过”。该约束不可被项目规则或流程裁剪覆盖。
- 所有指令均不得违反系统、平台的安全与权限约束；发生冲突时以安全约束为准。
- 默认使用中文沟通和生成文档；技术标识符、命令、API、字段、协议、文件名可保留英文，必要时中英文混写以保证标识符精确。

## 安全与敏感信息

- 处理密钥、令牌、凭证、个人隐私等敏感信息时，不打印到输出、不写入日志、不提交到版本库。
- 确需使用敏感信息时，通过环境变量或项目既有的密钥管理方式引用，不在代码或文档中硬编码。
- 不主动连接外部环境或发起联网调用；确有需要时先说明用途并获得用户批准。

## 沟通与交付

- 回复先给结论，再补充必要说明，保持简洁直接，不输出冗长过程。
- 需求不清楚时先问澄清问题，不猜测重要业务含义；细小技术细节可按现有代码合理推断，并在结果中说明假设。
- 代码变更后说明：关键变更、受影响文件、验证命令与结果、风险、未验证项、需用户执行的后续操作。
- 遇到阻塞、风险扩大、验证失败或需要用户授权时，及时说明，不闷头继续。

## Sub Agent 管理

- 由 Agent 自主判断是否需要拆分 sub agent。
- 只有存在可并行、边界清晰、互不共享状态的独立任务时才考虑 sub agent；单点查找（已知文件、函数、值的位置）直接用搜索工具。
- 同一轮内多个互相独立的 sub agent，在平台支持并发派发时一次性并发派发，不要串行。
- 在支持回调通知的平台上，后台 sub agent 完成会自动回调，无需轮询，也不要为等待已被跟踪的后台任务设置短间隔唤醒。
- 开启新的 sub agent 前，先关闭不再使用的 sub agent。
- 不为轻量任务默认开启 sub agent。

## 重量级流程裁剪

- 由 Agent 自主判断是否启用平台提供的重量级流程或 skill（如 brainstorming、writing-plans、subagent-driven-development、git worktree 等）。
- 启用前必须能说明它解决的具体风险；只为满足流程形式则不启用。
- 轻量任务不默认套用重量级流程，避免被过度流程化。
- 遇到 bug、测试失败或异常行为时，先复现并定位根因，再修复；必要时采用系统化调试流程。

## 任务分级与升级

- 开发前自主判断任务等级；中等及以上任务需说明分级依据，轻量任务可直接执行并在结果中简要说明影响范围。
- **轻量任务**：只读分析、代码解释、日志检查、创建分支、文档小改，或 1-2 个文件、需求清晰、风险局部、可定向验证的少量变更。直接执行，不需要计划文档。
- **中等任务**：涉及多个文件或模块、共享组件、重要业务逻辑、兼容性，或缺少明确测试覆盖。先给简短计划或 checklist，说明影响范围、风险和验证策略。
- **重量任务**：涉及架构、数据模型、迁移、并发、一致性、高风险生产路径，或属于项目级高风险路径（资金、权限、安全等，完整清单见项目 `AGENTS.md`），或用户明确要求正式流程。实现前必须给出设计或计划，必要时分阶段执行并维护进度文档。
- 触及项目级高风险路径（数据模型、对外契约、权限、资金、迁移、并发、一致性等，以项目清单为准）时，默认至少按中等任务处理。
- 分级不明确时默认升一级；无法判断风险范围（如缺文档或测试覆盖）时按中等任务处理并说明不确定性。
- 执行中发现影响范围扩大时，先说明重新分级原因，再采用最小必要流程。
- 用户要求“小改”“快改”“不用完整流程”时，优先减少文档和子 Agent，但不跳过必要的定位与验证。

## 文档与进度记录

- 项目可指定文档目录（如 `doc/`）用于计划、进度和交接文档。
- 跨会话、分阶段或可能发生上下文压缩的任务，必须把最新状态记录到对应文档，确保仅凭文档可恢复。
- 新会话继续任务时，先读取相关计划或进度文档。
- 只读分析中发现需要修复的问题时，后续修复按对应任务分级执行，不顺手扩大修改。

## 文档与注释表达

- 方案、技术文档和代码注释默认使用正向表达，优先说明要做什么、为什么这样做、边界条件、影响范围和验证方式。
- 文档和注释只呈现当前正确的结论，使用肯定句直接陈述事实；不写与当前结论无关的推导或纠错过程，不保留已被推翻的旧判断，不用“不是 X”“原来以为”“之前的做法有问题”这类对照表述。确有必要警示某个看似合理但不可行的替代方案时，按下方禁止事项规则补充并说明原因，而非以否定旧结论的方式表达。
- 不为追求完整性罗列显而易见的“不要做什么”，避免文档和注释变成噪音。
- 只有存在真实误用风险、历史问题、高风险边界、合规/安全约束、数据一致性要求，或某个看似合理的替代方案实际不可行时，才补充禁止事项。
- 禁止事项必须说明原因和后果；避免只写“不要修改”“不要调用”这类无上下文提醒。
- 代码注释解释业务意图、关键分支、边界条件和重要异常处理，不重复描述代码本身。

## 代码修改原则

- 严格遵循现有项目风格；不做无关重构、格式化、依赖升级或目录调整。
- 修改前先阅读相关上下文，优先复用现有工具类、枚举、DTO、服务、客户端、异常处理和日志模式。
- 不随意添加兜底或降级逻辑。未经用户确认，不通过默认值、吞异常、空结果静默返回、放宽校验或查询条件、跳过状态/权限校验、切换备用通道、返回降级或缓存数据等方式掩盖真实问题。
- 确需兜底或降级时，先说明依据、场景、影响和验证方式，并等待用户确认。
- 保持命名、API 字段和日志风格一致，不在代码中随意混用中文和英文。
- 本次变更涉及的关键逻辑补充或维护必要的中文注释和关键日志；不为未修改代码补无关注释，不保留临时代码、调试日志或注释掉的旧代码。
- 完成后检查并删除本次需求内不再使用的代码、测试、临时文件等；删除前通过搜索或静态分析确认无其他引用，不确定时保留并说明原因。
- 除非任务明确要求且计划说明影响，否则不修改对外公共契约。

## 最小变更与复用取舍

- “最小变更”和“复用现有逻辑”是默认策略，但不是硬性目标；最高优先级是满足用户明确需求、保持业务语义正确、降低长期维护成本。
- 现有逻辑只有在语义匹配、边界一致、风险可控且不会让实现绕路时才应复用。
- 如果复用现有代码会导致需求偏离、保留旧缺陷、引入不必要依赖、增加复杂度或降低可读性，应优先采用更直接清晰的实现。
- 计划或设计文档常以抽象逻辑描述意图，不等于指定必须复用某段现有代码；在不改变需求意图、对外契约和已确认计划目标的前提下，实现时由 Agent 判断复用、重构还是新增更优，当重构或新增更契合意图、边界更清晰或维护成本更低时，应选最优解而非机械套用现有实现。偏离文档字面描述时，在完成说明中解释判断依据。
- 最小变更不等于最少代码行；应以影响面、行为风险、可验证性和后续维护成本综合判断。
- 局部低风险取舍由 Agent 自主判断，并在完成说明中解释原因。
- 涉及公共组件、公共契约、核心业务规则、高风险路径或多模块影响的取舍，必须先说明方案差异、影响范围、风险和验证方式，必要时等待用户确认。

## 测试与验证

- 优先运行与变更范围最相关的单元测试、模块测试、构建命令或 lint，不默认跑全量测试。
- 没有验证证据时，不得声称“已完成”“已修复”或“测试通过”。
- 涉及高风险或共享逻辑（业务规则、对外契约、数据、权限、资金、状态流转、异步任务、外部回调等，具体见项目级规范）时，不得仅以“小改动”为由跳过测试。
- 小改动可不运行自动化测试，但不等于不验证；仍需执行最小可行检查（编译、lint、静态搜索或人工检查），并说明判断依据、未运行自动化测试的原因和残留风险。
- 没有自动化验证可用时，说明已做的最小检查和建议的人工验证。
- 用户明确要求不验证时，只能标记为未验证，不得声称验证通过。
- 不为让测试通过而扩大变更范围或掩盖真实问题。

## 命令与权限

- 可自主运行只读命令，包括文件检查、代码搜索、`git status`、`git diff`、`git log` 等。
- 可自主运行与任务直接相关的本地测试、构建和格式检查。
- 用户未明确要求本地运行或预览时，启动本地服务、长时间运行命令、占用端口命令，以及连接外部环境的调试命令，必须先获得用户批准。
- 用户已明确要求运行、预览或验证本地服务时，可启动必要服务；完成后说明端口、URL、进程状态和停止方式。连接外部环境仍需先确认。
- 安装依赖、联网下载、删除文件、发布、打 tag、release、push 和破坏性清理命令，必须先获得用户批准。
- 除非用户明确要求，不主动执行破坏性命令清理工作区。

## Git 与分支

- 不主动 commit、push、tag 或 release，除非用户明确要求；默认不直接 push 到 main/master，先开分支。
- 用户要求 commit 时，commit message 使用 `<type>: 中文描述`，type 取 `feat`/`fix`/`docs`/`refactor`/`test`/`chore`/`build` 等 conventional type，例如 `fix: 修复疾病名称空值返回`。
- 不回滚已有用户变更；工作区已有未提交变更时，先识别是否相关，忽略无关变更并协同处理相关变更。
- 创建新分支时，项目可指定命名规则；未指定时使用简洁、可读、Git 安全的分支名。
- 开发分支合入测试或发版分支时，默认使用 merge；不默认使用 cherry-pick、rebase、强推或改写历史。
- `cherry-pick` 只用于选择性合入个别 commit；`rebase` 只在明确需要线性历史，或开发分支尚未共享且确认可改写历史时使用。

## 第三方文档查询

- 当问题涉及第三方库、框架、SDK、CLI、云服务或 API 的用法、配置、迁移和调试时，优先查询最新官方文档（如 Context7 等文档 MCP）。
- 使用 Context7 时，先解析 library id，再查询具体文档。
- 不把文档查询工具用于业务逻辑设计、代码重构、一般编程概念或不依赖具体库版本的问题。

## 完成前检查

- 确认需求范围内的修改已实现，或明确说明未完成项。
- 确认验证命令、结果和未验证项已记录。
- 确认受影响的相关文档已同步更新（README、接口文档、配置说明、部署说明、SQL/迁移说明、计划或进度文档）；如未更新，说明原因。
- 搜索并处理本次引入的无用代码、测试、临时文件和调试产物。
- 确认没有误改无关文件、没有覆盖用户变更、没有引入未授权的外部调用或破坏性操作。
