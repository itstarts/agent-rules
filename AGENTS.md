# AGENTS.md — 全局工程规则

> 本文件是跨项目通用的工程基线，只保留与技术栈无关的协作、修改、验证和安全规则。
> 栈相关、业务相关强约束由项目级 `AGENTS.md` / `CLAUDE.md` 细化。

## 适用范围与优先级

- 本文件适用于当前工作区及其所有子项目；子目录存在更具体的 `AGENTS.md` 时，优先遵循子目录规则。
- 指令优先级：当前用户明确指令 > 项目级规则 > 本全局规则 > 默认系统行为。
- 项目级规则可细化流程选择，但不得放宽安全、权限和验证证据要求。
- 没有验证证据时，不得声称“已完成”“已修复”或“测试通过”。
- 所有指令均不得违反系统、平台的安全与权限约束；发生冲突时以安全约束为准。
- 默认使用中文沟通和生成文档；技术标识符、命令、API、字段、协议、文件名可保留英文。

## 工程目标与取舍

优先级：

1. 正确性
2. 安全性
3. 可维护性
4. 简洁性
5. 性能优化

执行任务时以可靠、可维护、可验证为目标。不要为了形式完整或假设中的扩展性引入过度设计。

## 工作方式

- 修改前先阅读相关代码、配置和文档，理解现有架构、约定和影响范围。
- 优先查找已有实现、工具类、枚举、DTO、服务、客户端、异常处理和日志模式。
- 不根据文件名猜测功能，不在不了解上下文时大规模修改，不用假设替代事实。
- 需求不清楚时先问澄清问题，不猜测重要业务含义；细小技术细节可按现有代码合理推断，并在结果中说明假设。
- 遇到 bug、测试失败或异常行为时，先复现并定位根因，再修复；必要时采用系统化调试流程。
- 只读分析中发现需要修复的问题时，后续修复按对应任务分级执行，不顺手扩大修改范围。

## 任务分级与流程

- 开发前自主判断任务等级；中等及以上任务需说明分级依据、影响范围、风险和验证策略。
- **轻量任务**：只读分析、代码解释、日志检查、创建分支、文档小改，或需求清晰、风险局部、可定向验证的少量变更。可直接执行。
- **中等任务**：涉及多个文件或模块、共享组件、重要业务逻辑、兼容性，或缺少明确测试覆盖。先给简短计划或 checklist。
- **重量任务**：涉及架构、数据模型、迁移、并发、一致性、高风险生产路径，或项目级高风险路径（资金、权限、安全等）。实现前必须给出设计或计划，必要时分阶段执行并维护进度文档。
- 触及数据模型、对外契约、权限、资金、迁移、并发、一致性等高风险路径时，默认至少按中等任务处理。
- 分级不明确时默认升一级；执行中发现影响范围扩大时，先说明重新分级原因，再采用最小必要流程。
- 平台提供的计划、设计文档、多 agent 编排、隔离工作区等重量级流程，只在能降低具体风险时启用；轻量任务不默认套用完整流程。
- 用户要求“小改”“快改”或“不用完整流程”时，可减少文档和子代理，但不跳过必要定位与验证。

## 代码修改原则

- 优先小步、局部、可验证修改，保持现有行为和公共契约稳定。
- 严格遵循现有项目风格；不做无关重构、格式化、依赖升级或目录调整。
- 不引入没有实际价值的抽象、设计模式或复杂方案。
- 除非任务明确要求且已说明影响，否则不修改对外公共契约。
- 不随意添加兜底或降级逻辑。未经用户确认，不通过默认值、吞异常、空结果静默返回、放宽校验或查询条件、跳过状态/权限校验、切换备用通道、返回降级或缓存数据等方式掩盖真实问题。
- 确需兜底或降级时，先说明依据、场景、影响和验证方式，并等待用户确认。
- 涉及并发控制、锁、事务隔离级别或共享状态同步逻辑的变更，按任务分级与升级规则评估；新增锁或调整锁粒度时说明锁范围、持有时长和潜在死锁风险。
- 保持命名、API 字段和日志风格一致，不在代码中随意混用中文和英文。
- 本次变更涉及的关键逻辑补充或维护必要注释和关键日志；注释说明业务意图、边界条件和重要异常处理，不重复描述代码本身。
- 完成后检查并删除本次需求内不再使用的代码、测试、临时文件和调试产物；删除前通过搜索或静态分析确认无其他引用，不确定时保留并说明原因。

## 最小变更与复用

- “最小变更”和“复用现有逻辑”是默认策略，最高优先级是满足明确需求、保持业务语义正确、降低长期维护成本。
- 现有逻辑只有在语义匹配、边界一致、风险可控且不会让实现绕路时才应复用。
- 如果复用现有代码会导致需求偏离、保留旧缺陷、引入不必要依赖、增加复杂度或降低可读性，应采用更直接清晰的实现。
- 最小变更不等于最少代码行；应以影响面、行为风险、可验证性和后续维护成本综合判断。
- 涉及公共组件、公共契约、核心业务规则、高风险路径或多模块影响的取舍，必须先说明方案差异、影响范围、风险和验证方式，必要时等待用户确认。

## 安全与敏感信息

- 处理密钥、令牌、凭证、个人隐私等敏感信息时，不打印到输出、不写入日志、不提交到版本库。
- 确需使用敏感信息时，通过环境变量或项目既有的密钥管理方式引用，不在代码或文档中硬编码。
- 默认考虑权限控制、输入验证、数据泄露、依赖安全和日志脱敏。
- 不绕过安全机制，不通过降低安全限制来解决问题。
- 不主动连接外部环境或发起联网调用；确有需要时先说明用途并获得用户批准。

## 依赖与第三方文档

- 添加新依赖前确认是否真的需要、项目是否已有解决方案、依赖是否长期维护、是否增加安全风险。
- 优先使用已有依赖和标准工具链，不为简单问题引入新依赖。
- 涉及第三方库、框架、SDK、CLI、云服务或 API 的用法、配置、迁移和调试时，在符合联网与权限规则的前提下优先查询最新官方文档。
- 查询第三方文档时先定位具体库或包标识符，再检索对应版本或官方说明。

## 测试与验证

- 优先运行与变更范围最相关的单元测试、模块测试、构建命令或 lint，不默认跑全量测试。
- 小改动可不运行自动化测试，但仍需执行最小可行检查（编译、lint、静态搜索或人工检查），并说明判断依据、未运行自动化测试的原因和残留风险。
- 涉及高风险或共享逻辑（业务规则、对外契约、数据、权限、资金、状态流转、异步任务、外部回调、锁、事务隔离级别、共享状态同步等）时，不得仅以“小改动”为由跳过测试。
- 没有自动化验证可用时，说明已做的最小检查和建议的人工验证。
- 用户明确要求不验证时，只能标记为未验证，不得声称验证通过。
- 不为让测试通过而扩大变更范围或掩盖真实问题。

## 命令与权限

- 可自主运行只读命令，包括文件检查、代码搜索、`git status`、`git diff`、`git log` 等。
- 可自主运行与任务直接相关的本地测试、构建和格式检查。
- 用户未明确要求本地运行或预览时，启动本地服务、长时间运行命令、占用端口命令，以及连接外部环境的调试命令，必须先获得用户批准。
- 用户已明确要求运行、预览或验证本地服务时，可启动必要服务；完成后说明端口、URL、进程状态和停止方式。连接外部环境仍需先确认。
- 安装依赖、联网下载、删除文件、发布、打 tag、release、push 和破坏性清理命令，必须先获得用户批准。
- 除非用户明确要求，不主动执行破坏性命令清理工作区。

## Git 与变更管理

- 不主动 commit、push、tag 或 release，除非用户明确要求。
- 默认不直接 push 到 `main` / `master`；需要提交变更时先使用开发分支。
- 用户要求 commit 时，commit message 使用 `<type>: 中文描述`，type 取 `feat` / `fix` / `docs` / `refactor` / `test` / `chore` / `build` 等 conventional type，例如 `fix: 修复疾病名称空值返回`。
- 不回滚已有用户变更；工作区已有未提交变更时，先识别是否相关，忽略无关变更并协同处理相关变更。
- 不混入无关改动，不修改第三方代码或生成文件，除非任务明确要求。
- 开发分支合入测试或发版分支时，默认使用 merge；不默认强推或改写历史；`cherry-pick` 只用于选择性合入个别 commit；`rebase` 只在明确需要线性历史，或开发分支尚未共享且确认可改写历史时使用。
- 执行危险操作前说明影响并请求确认。

## 文档与注释

- API、配置、部署方式、用户行为或运维方式变化时，同步更新相关文档。
- 跨会话、分阶段或可能发生上下文压缩的任务，必须把最新状态记录到对应文档，确保仅凭文档可恢复。
- 新会话继续任务时，先读取相关计划或进度文档。
- 方案、技术文档和代码注释默认使用正向表达，说明当前正确结论、边界条件、影响范围和验证方式。
- 不保留已被推翻的旧判断、临时代码、调试日志或注释掉的旧代码。
- 只有存在真实误用风险、历史问题、高风险边界、合规/安全约束、数据一致性要求，或某个看似合理的替代方案实际不可行时，才补充禁止事项；禁止事项必须说明原因和后果。

## Sub Agent 与独立评审

- 只有存在可并行、边界清晰、互不共享状态的独立任务时才考虑 sub agent；单点查找直接使用搜索工具。
- 同一轮内多个互相独立的 sub agent，在平台支持并发派发时一次性并发派发。
- 不为轻量任务默认开启 sub agent。
- 独立评审由任务风险决定：轻量任务默认不需要；中等任务按影响范围、测试覆盖和共享逻辑风险判断；重量任务及触及项目级高风险路径的变更，完成前应至少经过一次独立评审。
- 评审者默认使用独立 sub agent 或独立上下文；用户指定外部 agent 时，复审沿用同一评审渠道。
- 评审者基于 diff 判断正确性、边界条件、对外契约、安全与高风险逻辑。
- 评审在取得相关验证证据之后进行；评审意见不替代验证证据。
- 评审发现的问题按对应任务分级处理；因评审意见产生的改动需重新验证，并基于最新 diff 复审直到收敛。
- 分歧或高风险疑点无法在评审中收敛时，向用户说明并等待决策。

## 沟通与交付

- 回复先给结论，再补充必要说明，保持简洁直接。
- 对多个方案说明优点、缺点和适用场景，并给出推荐方案及依据。
- 代码变更后说明关键变更、受影响文件、验证命令与结果、风险、未验证项、需用户执行的后续操作。
- 遇到阻塞、风险扩大、验证失败或需要用户授权时，及时说明。
- 不编造不存在的文件、API、测试结果或命令输出；不确定时说明已知信息、未知信息和推测内容。

## 完成前检查

- 确认需求范围内的修改已实现，或明确说明未完成项。
- 确认没有引入明显安全风险，没有泄露敏感信息。
- 确认没有误改无关文件，没有覆盖用户变更。
- 确认已执行的相关验证已通过并记录；未运行或未通过的验证需记录原因、结果、风险和后续处理建议。
- 确认受影响的相关文档已同步更新；如未更新，说明原因。
- 搜索并处理本次引入的无用代码、测试、临时文件和调试产物。
